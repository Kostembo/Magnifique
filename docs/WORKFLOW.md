# Рабочий процесс с агентами

## Принцип

Один агент не должен одновременно планировать, писать и оценивать свою же работу.
Разделяем на три роли: **Архитектор → Исполнитель → Ревьюер**.

Каждая роль — отдельная сессия (или явный переключатель контекста).

---

## Роль 1: Архитектор

**Когда:** перед началом любой нетривиальной задачи (новый модуль, изменение схемы БД, новый API-маршрут).

**Задача:** составить план. Кода не писать.

**Что выдаёт:**

```
## План: [название задачи]

### Какие файлы затрагиваем
- src/app/api/X/route.ts — создать новый маршрут
- prisma/schema.prisma — добавить поле Y (если нужно)
- src/app/(dashboard)/X/page.tsx — обновить UI

### Логика
1. Шаг 1: ...
2. Шаг 2: ...

### Риски
- Изменение схемы требует миграции
- Нужно проверить роль в эндпоинте

### Что НЕ трогаем
- src/lib/auth.ts — без изменений
- компоненты не из этого модуля
```

**Промт для архитектора:**
> "Ты Архитектор. Задача: [описание]. Прочитай docs/ARCHITECTURE.md и docs/DATA_MODEL.md. Составь план: какие файлы создать/изменить, какая логика, какие риски. Кода не пиши."

---

## Роль 2: Исполнитель

**Когда:** есть готовый план от Архитектора.

**Задача:** реализовать строго по плану. Не расширять скоуп.

**Правила:**
- Читает CLAUDE.md перед началом.
- Если нужно отступить от плана — сначала сообщает, не делает молча.
- После реализации прогоняет `npm run lint`.
- Коммит только после прохождения lint.

**Промт для исполнителя:**
> "Ты Исполнитель. Вот план от Архитектора: [план]. Реализуй строго по плану. Прочитай CLAUDE.md — соблюдай все запреты. После реализации запусти lint."

---

## Роль 3: Ревьюер

**Когда:** задача реализована, перед тем как считать её завершённой.

**Задача:** найти проблемы, которые исполнитель не видит у себя — свежий контекст.

**Что проверяет:**
1. Безопасность — чеклист из `docs/SECURITY.md`
2. Нарушения правил из `CLAUDE.md` (запреты, паттерны)
3. Дублирование кода / нарушение DRY
4. Edge cases — что сломается при граничных значениях
5. Типизация — нет ли `any`, корректны ли типы

**Промт для ревьюера:**
> "Ты Ревьюер. Вот изменения: [диф или список файлов]. Проверь: 1) безопасность по docs/SECURITY.md, 2) нарушения CLAUDE.md, 3) дублирование, 4) что может сломаться. Не объясняй что код делает — только проблемы."

---

## Схема сессии

```
[ Сессия 1: Архитектор ]
  Вход: описание задачи + docs/
  Выход: план в виде текста

[ Сессия 2: Исполнитель ]
  Вход: план + CLAUDE.md
  Выход: рабочий код + lint OK + коммит

[ Сессия 3: Ревьюер ]
  Вход: диф (изменения) + CLAUDE.md + docs/SECURITY.md
  Выход: список проблем (если есть) или "OK"

[ При проблемах → Сессия 4: Исполнитель ]
  Вход: список проблем от Ревьюера
  Выход: исправления + новый коммит
```

---

## Когда роль Архитектора можно пропустить

Для небольших задач (баг-фикс, изменение текста, мелкий UI) Архитектор не нужен.
Правило: **если задача затрагивает > 2 файлов или меняет БД/API — нужен Архитектор**.

---

## Спека до кода — всегда

Перед новым модулем (Фаза N) — одна сессия только на документы:
1. Обновить `docs/PRODUCT.md` — что входит в эту фазу
2. Обновить `docs/DATA_MODEL.md` — какие новые модели/поля
3. Создать `docs/SPEC_PHASE_N.md` — детальная спецификация фичи
4. Только после — код

Документы живут в репозитории и являются **источником правды** для всех агентов.
